DGOP 
Planning Assumptions
Sprint length: 2 weeks.
Delivery length: 36 sprints, aligned to the original 18-month roadmap.
Architecture: Angular frontend, Node.js backend, local PostgreSQL for development.
Recommended backend framework: NestJS with TypeScript, because the DGOP domain is large and benefits from modules, dependency injection, guards, DTO validation, OpenAPI generation, and testability.
Recommended ORM: Prisma or TypeORM. Prisma is recommended for fast schema iteration, migrations, type-safe access, and local PostgreSQL development.
Recommended frontend structure: Angular standalone components or feature modules with Angular Material, routing, guards, interceptors, reactive forms, and a shared design system.
Recommended repo shape: monorepo with apps/web, apps/api, and shared packages for DTOs, types, constants, and validation schemas.
First implementation style: modular monolith backend. Keep clear module boundaries so services can be split later if needed.
Local file storage can be used first for evidence uploads; object storage such as MinIO can be added later.
External integrations start as mock adapters, CSV import/export, and documented API contracts before connecting to real tools.
UI/UX is a first-class, non-negotiable requirement. Every screen in every sprint must use one shared, professional, interactive, accessible, and consistent design system. No screen ships with ad hoc styling, one-off components, or inconsistent patterns.
The app is published externally from Sprint 0. The server is temporarily hosted on the developer laptop and reachable from anywhere over HTTPS, so the user can test each increment remotely.
After every sprint, the deliverables include user stories and test cases so the user can validate the increment.
UI/UX Design System And Experience Standards
This section is binding for every sprint and every screen. The goal is a professional, modern, highly interactive product with a single consistent look, feel, and behavior across all modules.
UX Principles
Consistency first: identical patterns for lists, detail pages, forms, wizards, dashboards, tables, filters, and dialogs across all modules.
Clarity over decoration: governance users scan large data sets, so prioritize legibility, hierarchy, scannability, and meaningful density.
Progressive disclosure: show summary first, then detail on demand using tabs, drawers, expandable rows, and Asset 360 style panels.
Guided workflows: multi-step actions (assessment wizards, approvals, evidence submission) use clear stepper patterns with validation and a visible progress state.
Feedback always: every action gives immediate, visible feedback through loading, success, warning, error, and empty states.
Forgiving by design: confirmations for destructive actions, undo where feasible, autosave or draft state for long forms, and no dead ends.
Accessibility is mandatory, not optional.
Design Tokens (Single Source Of Truth)
Define design tokens in shared theme files: color, typography scale, spacing scale, radius, elevation/shadow, border, z-index, motion duration, and easing.
Tokens drive both Angular Material theming and custom components so visuals never drift.
Semantic color tokens, not raw hex in components: surface, on-surface, primary, secondary, success, warning, danger, info, muted, plus governance status colors mapped to AppDocs (Green on track, Amber at risk, Red overdue/non-compliant, Grey not started, Blue under review).
Light and dark themes from day one, switchable, both meeting contrast requirements.
Typography supports Latin and Arabic with a font stack that renders Arabic cleanly.
Component Library (Reusable, Built Once)
Build a shared UI library in packages/ui (or apps/web/src/app/shared/ui) and reuse it everywhere:
Layout: app shell, responsive sidebar, top bar, breadcrumbs, page header with title/actions, content sections, split views, drawers.
Data display: standardized data table (sort, multi-filter, column show/hide, pagination, density toggle, sticky header, row selection, bulk actions, export), key-value detail panels, timelines, status chips/badges, KPI cards, charts, tag inputs.
Inputs: text, textarea, select, multi-select, async lookup/autocomplete, date/date-range (Gregorian and Hijri aware), number, toggle, radio, checkbox, file upload with drag-and-drop and progress, rich validation messages.
Navigation: tabs, stepper/wizard, pagination, command palette/global search.
Feedback: toasts/snackbars, inline alerts, confirmation dialogs, full and skeleton loaders, progress bars, empty states, error states with retry.
Overlays: modal dialog, side drawer/sheet, popover, tooltip, context menu.
All components are themeable via tokens, fully keyboard operable, and documented with usage examples.
Interaction And Motion
Subtle, purposeful motion only: 150-250ms transitions for hover, focus, expand/collapse, drawer/dialog open, and page/route transitions.
Respect prefers-reduced-motion and disable non-essential animation when requested.
Optimistic UI where safe, with clear rollback on failure.
Interactive tables and dashboards: client-side responsiveness plus server-side pagination/filtering for large data.
Keyboard support: full tab order, focus rings, shortcuts for common actions (save, search, new), and accessible focus trapping in dialogs.
Accessibility (WCAG 2.1 AA Target)
Color contrast meets AA for text and UI components in both themes.
All interactive elements are reachable and operable by keyboard.
Correct semantic roles, labels, and ARIA where needed; visible focus states.
Form fields have associated labels, descriptions, and accessible error text.
No information conveyed by color alone; pair status color with icon and text.
Screen-reader friendly tables, dialogs, steppers, and live regions for async updates.
Bilingual And RTL (Arabic And English)
Full RTL and LTR support driven by language selection, including layout mirroring, icon direction, and number/date formatting.
All labels come from i18n resource files; no hardcoded strings in components.
Components validated in both directions every sprint (no truncation, overflow, or broken alignment).
Responsive And Cross-Device
Mobile-first responsive breakpoints; layouts adapt for phone, tablet, and desktop.
Tables degrade gracefully (horizontal scroll, priority columns, or card view on small screens).
Touch targets meet minimum size; no hover-only interactions without a non-hover equivalent.
Consistency Governance
A living UI kit / component showcase page (Storybook or an in-app "Design System" route) documents every component and pattern.
Visual regression and accessibility checks run in CI (for example Storybook + axe + screenshot diffing) so consistency does not regress.
Definition of Done includes a UI/UX consistency check for every screen.
New patterns are added to the design system first, then consumed; one-off styling is not allowed.
External Publishing And Remote Access (From Sprint 0)
The app must be reachable from anywhere starting in Sprint 0, while the server runs temporarily on the developer laptop. This lets the user test every sprint remotely without deploying to a cloud host yet.
Approach
Run the Node.js API and the Angular app locally on the laptop using the .env port settings.
Expose them to the internet through a secure HTTPS tunnel rather than direct port forwarding, because tunnels avoid router/NAT/firewall changes, provide TLS automatically, and give a stable public URL.
Serve the app over a single public hostname. The Angular app calls the API under the same hostname using a path prefix such as /api to avoid cross-origin and certificate issues.
Recommended Tooling (pick one, default first)
Cloudflare Tunnel (recommended default): free, stable named tunnel, automatic HTTPS, optional custom domain, and Cloudflare Access for adding an authentication gate in front of the app. Good for an always-on laptop demo URL.
ngrok: fastest to start; free tier gives a random HTTPS URL, paid gives a reserved domain. Good for quick sessions.
Tailscale Funnel: simple HTTPS exposure tied to a Tailscale identity; good if access can be limited.
The plan defaults to Cloudflare Tunnel for a persistent URL and an optional access gate, with ngrok as a quick fallback. The chosen tool is documented in the runbook and can be swapped without code changes.

Security Requirements For Public Exposure (Mandatory From Sprint 0)
Exposing a laptop publicly is sensitive, so these controls are required from the first sprint:
HTTPS only; no plain HTTP exposure.
Authentication required before any data is shown; no anonymous access to app data.
Optional extra gate (for example Cloudflare Access, basic auth at the tunnel, or an IP/email allowlist) during early sprints when only invited testers should reach the app.
Strict CORS limited to the public app origin.
Security headers, rate limiting, and brute-force protection on auth endpoints from Sprint 0.
Do not load real personal or sensitive (PDPL/classified) data onto the laptop; use synthetic or anonymized seed data for remote testing.
Keep .env secrets off the public surface and out of version control; never expose the database port through the tunnel (tunnel the app only, not PostgreSQL).
Be able to take the tunnel down quickly; treat the public URL as temporary and revocable.
Log access and auth attempts so remote testing activity is auditable.
Operational Notes
Provide simple scripts/runbook steps to start the API, serve the web app, and start the tunnel.
Publish a short "How to access this sprint" note with the URL and test credentials for each sprint.
Because the host is a laptop, availability is best-effort during this phase; production hosting, backups, and uptime are addressed later in the roadmap.
Per-Sprint QA Deliverables (User Stories And Test Cases)
After every sprint (starting with Sprint 0), produce a sprint test package so the user can validate the increment remotely.
What Is Delivered Each Sprint
User stories for everything built in the sprint, in the format: "As a [role], I want [capability], so that [value]," each with clear acceptance criteria.
Test cases mapped to those user stories, each with: ID, title, related user story, preconditions, test data, step-by-step actions, expected result, and a pass/fail field.
A mix of positive, negative, validation, permission/role, and (where relevant) bilingual/RTL and accessibility checks.
The public test URL and test account(s) per role needed to run the tests.
A short release note summarizing what changed and any known limitations.
Where It Lives
Store the package under a QA/ folder (for example QA/Sprint-00/user-stories.md and QA/Sprint-00/test-cases.md) so it is versioned with the code.
Keep a consistent template so each sprint's package looks the same and is easy to execute.
Example Format
User story example:
US-00-01: As an administrator, I want to open the published app over HTTPS from a remote device,
so that I can confirm the environment is reachable before features are built.
Acceptance: The public URL loads over HTTPS, shows the app shell, and is not accessible over plain HTTP.
Test case example:
TC-00-01 | Remote HTTPS access
Linked: US-00-01
Preconditions: Tunnel running; laptop server up.
Steps: 1) Open the public URL on a remote device. 2) Confirm the padlock/HTTPS. 3) Try the http:// URL.
Expected: HTTPS loads the app shell; http:// redirects to HTTPS or fails; no data shown before login.
Result: Pass/Fail

Definition Of Done For Every Sprint
Every sprint should finish with:
Backend APIs implemented and documented.
Angular screens implemented with validation and role-aware behavior.
PostgreSQL migrations committed.
Seed data updated where needed.
Unit tests for critical services.
API tests for main endpoints.
Frontend tests for key components or flows.
Audit logging added for important changes.
Access rules checked.
UAT notes or demo script written.
Known issues logged.
No secret values committed.
The sprint increment is deployed to the laptop server and verified reachable from a remote device over the public HTTPS URL.
User stories for the sprint are delivered under QA/Sprint-XX/user-stories.md.
Test cases for the sprint are delivered under QA/Sprint-XX/test-cases.md, with a public URL and per-role test accounts so the user can execute them remotely.
UI/UX Definition of Done (mandatory for every screen in every sprint):
Built only from the shared design system and component library; no one-off components or ad hoc styling.
All visuals use design tokens; light and dark themes both correct.
Consistent patterns for lists, detail pages, forms, wizards, dialogs, and dashboards.
Loading, empty, success, warning, and error states implemented for every async view.
Full keyboard operability, visible focus, and accessible labels/roles; WCAG 2.1 AA met (verified with axe).
Status conveyed by color plus icon/text, mapped to AppDocs status colors.
RTL (Arabic) and LTR (English) parity verified; all strings from i18n files, no hardcoded text.
Responsive across phone, tablet, and desktop.
New or changed components documented in the design system showcase.
Visual regression and accessibility checks pass in CI.
Cross-Cutting Engineering Standards
Backend standards:
Use DTO validation for all input.
Use consistent error responses.
Use transactions for multi-step state changes.
Use soft deletes for governance records.
Use UUID identifiers.
Add created/updated/deleted audit fields.
Keep business rules in services, not controllers.
Keep integration logic behind adapters.
Frontend standards:
Use route guards for protected features.
Use reactive forms for complex forms.
Use shared components for lookup fields, status badges, tables, and timelines.
Use lazy-loaded feature areas.
Use role-aware navigation and action visibility.
Support Arabic and English labels where the data model requires bilingual fields.
Keep API calls in feature services.
UI/UX standards (consistency is mandatory):
One design system is the single source of truth; all screens consume shared tokens and components. One-off styling and duplicate components are not allowed.
Drive all visuals from design tokens (color, type, spacing, radius, elevation, motion); never hardcode colors or sizes in components.
Reuse the standardized data table, forms, dialogs, drawers, steppers, KPI cards, charts, status chips, toasts, and loaders everywhere.
Implement loading, empty, success, warning, and error states for every async view.
Meet WCAG 2.1 AA: keyboard operability, visible focus, semantic roles/labels, sufficient contrast, and no color-only meaning (pair with icon/text).
Support light/dark themes and full RTL/LTR with i18n; no hardcoded strings.
Use subtle, purposeful motion (150-250ms) and respect prefers-reduced-motion.
Be responsive and mobile-friendly; provide non-hover equivalents for all hover interactions.
Document every component/pattern in the design system showcase and protect consistency with visual regression and accessibility checks in CI.
Database standards:
Use migrations for every schema change.
Add indexes for filters used by dashboards and search.
Use foreign keys for core relationships.
Use JSON fields only when flexible metadata is required.
Keep evidence file metadata in the database, not only the file system.
Add audit records for important state changes.
Security standards:
Never commit .env secrets.
Add .env.example with placeholders.
Hash passwords for local users.
Restrict evidence access by role and context.
Log evidence upload, download, approval, rejection, and deletion attempts.
Use server-side authorization even when frontend hides controls.
Validate file uploads by type, size, and ownership.

Key Risks And Mitigations
Scope is large: deliver by releases and avoid building advanced modules before foundation workflows and evidence are stable.
NDI spreadsheet is broad: import specs early, but automate seven core domains first.
External systems may not be ready: use adapter interfaces, CSV import, and mock REST endpoints first.
Security requirements are significant: add audit and authorization from Sprint 1 rather than after features are built.
Evidence files are compliance-critical: store metadata, hashes, access logs, and status history from the first evidence sprint.
Local PostgreSQL is fine for development: production planning should later include backups, migration strategy, retention, and secure storage.
UI/UX consistency can drift across a large app: mitigate by building the design system first, consuming shared components only, enforcing the UI/UX Definition of Done, and running visual regression plus accessibility checks in CI every sprint.
Publishing a laptop to the internet is a security and availability risk: mitigate with HTTPS-only tunneling (never expose the database), authentication and an optional access gate, strict CORS, security headers, rate limiting, synthetic/anonymized data only, auditable access logs, and the ability to revoke the public URL quickly. Laptop hosting is temporary and best-effort; production hosting and uptime come later in the roadmap.
Per-sprint QA load is real: mitigate by using a fixed user-story and test-case template under QA/, seeding per-role test accounts, and keeping the test package scoped to the sprint increment.
