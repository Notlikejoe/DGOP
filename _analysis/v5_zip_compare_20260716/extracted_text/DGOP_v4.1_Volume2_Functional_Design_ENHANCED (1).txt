1. DGOP Administration Module
The DGOP Administration Module is the configuration backbone of the platform. It enables administrators to define the complete governance framework without code changes. All configuration is version-controlled, auditable, and subject to approval workflows.
1.1 Domain Management
Data domains are the primary classification of enterprise data independent of organizational structure. The Domain Management function provides: Domain Registry: Create, edit, and retire data domains with unique codes, bilingual names (Ar/En), descriptions, and hierarchical parent-child relationships. Domain Hierarchy: Visual tree editor for managing multi-level domain hierarchies (e.g., Customer -> Customer Profile -> Customer Interaction -> Customer Complaints). Default Ownership: Assign default domain owner and steward for automatic inheritance when assets are mapped to the domain. Domain Committee: Link domains to governance committees and working groups. NDI Mapping: Associate domains with relevant NDI domains and criteria. Status Lifecycle: Draft -> Active -> Retired with workflow-driven approval for status changes.
Key Attributes:
1.2 Subject Management
Data subjects define who or what the data is about. Subject management is critical for PDP, privacy, DLP, and retention governance. Subject Registry: Define subject types with bilingual names, personal data flags, sensitivity flags, and legal basis requirements. Privacy Configuration: Set consent requirements, DPIA triggers, retention rules, and cross-border restrictions per subject. Default Stewardship: Assign default privacy owner and business steward for each subject type. Subject Hierarchy: Support hierarchical subjects (e.g., Employee -> Contractor -> Intern) with inherited privacy settings.
Key Attributes:
1.3 Capability Management
Business capabilities define what the organization does with data. Capability management links processes, data assets, systems, and governance accountability. Capability Registry: Define capabilities with bilingual names, process owners, criticality levels, and parent-child hierarchy. Capability Mapping: Link capabilities to data domains, data subjects, systems, reports, and KPIs. Ownership by Capability: Assign owners and stewards based on capability rather than department, enabling cross-functional governance. NDI Alignment: Map capabilities to NDI domains for compliance tracking.
1.4 System Management
Systems represent the technology platforms that create, store, or process data. System management provides the technical context for stewardship. System Registry: Catalog all systems with unique codes, bilingual names, system types (ERP, CRM, DWH, etc.), and technical stewards. System Classification: Tag systems with data residency, criticality, and security zone. Asset Linkage: Systems are linked to data assets for lineage and impact analysis. Integration Endpoints: Store system API endpoints and connection details for DGOP integration.
1.5 Organization Unit Management
Organization units represent the formal hierarchy of the entity. This dimension is used for departmental ownership and reporting. Org Registry: Define departments, divisions, sections, units, and committees with bilingual names and manager assignments. Hierarchy Management: Visual org chart editor with drag-and-drop restructuring. Person Assignment: Link people to org units for automatic ownership inference. Historical Tracking: Track org changes over time for historical ownership analysis.
1.6 Role & RACI Management
Roles define the governance personas that participate in DGOP activities. RACI templates define how these roles interact for each activity. Role Registry: Define roles with codes, names, descriptions, and mandatory training requirements. Role Hierarchy: Support role inheritance (e.g., Senior Steward inherits from Steward). RACI Template Library: Pre-defined RACI templates for common governance activities (asset creation, classification, sharing, OD publication, FOI response). Custom RACI: Administrators can create custom RACI matrices for entity-specific activities. * Role Permissions: Map roles to DGOP functional permissions and data access rights.

For RACI matrix configuration with validation rules and RACI template library management, see Module 20.5 (RACI Management).

1.7 Assignment Rule Engine
The Assignment Rule Engine is the intelligent core that determines ownership and stewardship based on dimensional context. Rule Editor: Visual rule builder with conditions (AND/OR), operators (equals, contains, in list), and role assignments. Condition Types: Domain, Subject, Capability, Org Unit, System, Classification, Personal Data Flag, Sensitive Data Flag. Priority System: Rules evaluated in priority order (1-999). Lower numbers = higher priority. Conflict Detection: Automatic flagging when multiple rules match with different outcomes. Simulation Mode: Test rules against sample assets before activation. Version Control: Rules are versioned with effective dates and change history. * Approval Workflow: Rule changes require DMO Admin approval before activation.
1.8 Workflow Template Management
Workflow templates define the BPMN process models for governance activities. Administrators can configure templates without coding. Template Library: Pre-built templates for Owner Assignment, Steward Assignment, DQ Remediation, DLP Review, Data Sharing, OD Publication, FOI Response, DPIA Approval, Access Review, Evidence Review, and Appeals. Visual Editor: Camunda Modeler integration for BPMN 2.0 diagram creation and editing. Step Configuration: Define user tasks, service tasks, timer events, gateways, and decision logic. Role Assignment: Map each workflow step to DGOP roles for automatic task routing. Variable Mapping: Define process variables and their mapping to DGOP data entities. Versioning: Workflow templates are versioned with migration paths for in-flight cases.
[NEW v4.1] For comprehensive workflow configuration including workflow activity configuration, sequence configuration, approval path configuration, and workflow version management with impact analysis, see Module 20.1 (Workflow Configuration). For the full workflow execution engine with state machine transitions, parallel/sequential execution, and progress tracking, see Module 20.2 (Workflow Execution Engine). For workflow monitoring capabilities including status dashboards, timeline views, and performance metrics, see Module 20.10 (Workflow Monitoring).
1.9 SLA Template Management
SLA templates define time-based service levels for workflow tasks and governance activities. SLA Definition: Define SLA durations in business days, hours, or minutes with KSA calendar support. Escalation Rules: Define escalation paths when SLA is breached (notify manager, notify DMO, notify Data Council, notify DGSC). Calendar Configuration: Exclude weekends (Friday/Saturday) and KSA public holidays from SLA calculation. Warning Thresholds: Configure early warning alerts at 50%, 80%, and 95% of SLA elapsed. * KPI Linkage: SLA compliance is automatically fed into steward scorecards and NDI readiness calculations.
2. Ownership & Stewardship Registry
The Ownership & Stewardship Registry is the official system of record for data governance accountability. It maintains every assignment, delegation, approval, and performance metric related to data ownership and stewardship.
2.1 Assignment & Approval
Ownership assignment is not automatic – it is a governed workflow with validation, approval, and audit. Assignment Request: Triggered by asset creation, catalog sync, rule engine recommendation, or manual request. Rule Validation: The Assignment Rule Engine evaluates the asset against all active rules and recommends owner/steward. Conflict Check: If multiple rules match or no rule matches, the case is flagged for DMO review. Approval Chain: Recommended assignment is routed to the proposed owner for acceptance, then to the DMO for certification. Certification: Once approved, the assignment is published to the registry and written back to the data catalog. Notification: All parties (owner, steward, DMO, affected users) are notified of the new assignment.
Assignment Record:
2.2 Delegation & Backup
Governance cannot stop when an owner or steward is unavailable. DGOP provides robust delegation and backup mechanisms. Delegation: An owner or steward can delegate their responsibilities to another qualified individual for a defined period. Delegation requires approval from the DMO and notification to the Data Council. Backup Steward: Every primary steward must designate a backup steward who is automatically activated if the primary steward is on leave, transferred, or terminated. Temporary Assignment: For projects or emergencies, temporary stewards can be assigned with automatic expiry. Delegation Audit: All delegation records are maintained with full audit trails including reason, duration, approver, and outcomes. * Auto-Escalation: If a task is not completed within SLA and the assignee has an active delegation, the task is automatically routed to the delegatee.
2.3 Approval History
Every assignment, change, and revocation is recorded in the Approval History with full evidentiary detail. Change Tracking: Before/after states for every field change (owner, steward, effective dates, status). Approval Chain: Complete record of who requested, who reviewed, who approved, and who was notified. Comments & Justification: All approvals require comments explaining the decision. Evidence Attachment: Supporting documents (emails, meeting minutes, committee decisions) can be attached to approval records. * Reversion: Approved assignments can be reverted with full audit trail if errors are discovered.
2.4 Steward Workload Management
The DMO must balance stewardship workload to prevent burnout and ensure responsiveness. Workload Dashboard: Real-time view of open tasks, overdue items, and SLA compliance per steward. Capacity Planning: Track average tasks per steward, identify overloaded individuals, and recommend rebalancing. Workload Alerts: Automatic alerts when a steward's open task count exceeds configurable thresholds. Reassignment: DMO can reassign tasks from overloaded stewards to backups or peers with workflow-driven approval. * Trend Analysis: Historical workload trends inform staffing decisions and training priorities.
2.5 Effective Date Management
Governance assignments are not static – they have lifecycles with effective dates, expiry dates, and renewals. Effective Date: The date from which the assignment is active. Can be future-dated for planned transitions. Expiry Date: The date after which the assignment automatically expires. Used for temporary assignments, project-based stewardship, and term-limited roles. Renewal Workflow: Before expiry, DGOP triggers a renewal workflow to the owner/steward and DMO for extension or replacement. Grace Period: Configurable grace period after expiry during which the assignment remains active pending renewal. * Historical View: Full historical view of all past assignments for any asset, domain, or subject.
3. Data Asset Governance Hub
The Data Asset Governance Hub provides a single, unified view of every data asset enriched with governance metadata from DGOP and integrated tools. It is the operational command center for data stewards, owners, and DMO staff.
3.1 Asset 360 View

10. RCA Technique Integration [NEW v3.1]
The Root Cause Analysis (RCA) Technique Integration module provides structured investigation templates and guided workflows for identifying the true root cause of data quality issues. Rather than treating symptoms, this module ensures stewards apply rigorous analytical techniques to eliminate underlying causes.
10.1 5 Whys Analysis
The 5 Whys technique iteratively asks "Why?" to drill down from symptom to root cause.
Methodology
Define the Problem: Clearly state the observable data quality issue.
Ask Why (1st): Why did the problem occur?
Ask Why (2nd): Why did the answer to #1 happen?
Ask Why (3rd): Why did the answer to #2 happen?
Ask Why (4th): Why did the answer to #3 happen?
Ask Why (5th): Why did the answer to #4 happen?
Identify Root Cause: The 5th (or final) answer reveals the root cause.
Define Corrective Action: Action to eliminate the root cause.
DGOP Implementation
Guided Form: Step-by-step input form with "Why?" prompts and answer fields.
Validation: Minimum 3 whys required; maximum 7 whys supported.
Branching Support: If multiple causes identified at any level, branch into parallel tracks.
Evidence Linkage: Each "Why" answer can be linked to supporting evidence (logs, screenshots, system records).
Root Cause Classification: Root cause categorized using standard taxonomy (see Section 10.5).
Corrective Action Linkage: Direct link from root cause to corrective action in remediation plan.

12. MDM Matching & Survivorship Engine [NEW v3.1]
The MDM Matching & Survivorship Engine provides comprehensive master data management capabilities for identifying, matching, resolving, and consolidating duplicate records across systems. It supports deterministic and probabilistic matching algorithms, configurable survivorship rules, and golden record management for seven master data domains.
12.1 Matching Algorithms
The engine supports both deterministic and probabilistic matching approaches:
Deterministic Matching
Deterministic matching uses exact or standardized comparisons on specified match keys.
Probabilistic Matching
Probabilistic matching uses statistical models to compute match probability across multiple fields.
Match Score Calculation
Match Score = SUM(Field Weight * Field Match Score) / SUM(Field Weights)
12.2 5-Step Match Resolution
All potential matches follow a structured 5-step resolution process:
Step 1: Candidate Identification
System identifies potential matches using blocking keys (high-confidence match attributes).
Blocking reduces comparison space from O(n^2) to manageable subsets.
Candidate pairs/triples generated with initial match scores.
Step 2: Scoring & Classification
Full match score calculated using configured algorithm and weights.
Each candidate pair classified as Definite, Probable, Potential, or Non-Match.
Classification determines routing for subsequent steps.
Step 3: Auto-Resolution
Definite matches (score >= threshold): Automatically merged based on survivorship rules.
Probable matches (medium-high score): Auto-merged with DQ flag for post-merge review.
Potential matches: Routed to manual review queue.
Non-matches: Discarded, record kept as unique.
Step 4: Manual Review
DQ Steward or Data Steward reviews potential matches in a side-by-side comparison view.
For each candidate pair, steward selects: Match, Not Match, or Needs Investigation.
Review assisted by:
Side-by-side field comparison highlighting differences
Match score breakdown per field
System recommendations based on historical decisions
Related records context (e.g., other family members for customer matching)
Bulk review supported for high-volume scenarios.
Step 5: Merge & Survivorship Application
Upon Match confirmation: Source records are merged into a golden record.
Survivorship rules determine which field values from which source record are retained.
Merge history preserved for audit.
Source record links maintained (golden record knows its sources).
Downstream systems notified of merge event.

15. Data Masking Service [NEW v3.1]
The Data Masking Service provides six data masking techniques to protect sensitive data across different environments and use cases. It enables automated, policy-driven masking based on data classification and environment type, ensuring that sensitive data is never exposed in non-production contexts while preserving data utility for testing, development, and analytics.
15.1 Six Masking Techniques
1. Static Masking
Static masking permanently replaces sensitive data with masked values in a copy of the database.
2. Dynamic Masking
Dynamic masking applies masking rules in real-time at query time based on user identity and context.
3. Tokenization
Tokenization replaces sensitive data with non-sensitive tokens that maintain referential integrity.
4. Anonymization
Anonymization irreversibly removes all identifying information such that the data subject cannot be re-identified.
5. Pseudonymization
Pseudonymization replaces identifying fields with pseudonyms while maintaining a separate re-identification key.
6. Redaction
Redaction permanently removes or obscures specific portions of data.
