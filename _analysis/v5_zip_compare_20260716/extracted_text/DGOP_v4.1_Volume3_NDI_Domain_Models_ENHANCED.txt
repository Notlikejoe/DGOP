1. Introduction to NDI Domain Operationalization
1.1 Purpose
This volume defines the operational models for every NDI domain within the DGOP ecosystem. Each operational model translates NDI specifications into executable workflows, evidence artifacts, KPIs, and compliance controls.
v4.1 Enhancement – DSP Integration: Each domain now explicitly integrates Operational Data Steward (ODS), Project Data Steward (PDS), and Enterprise Data Steward (EDS) roles into operational lifecycles, applies the 4-tier severity-based DQ SLA framework, tracks evidence quality across 6 DQ dimensions, and embeds Privacy by Design controls where applicable.
v4.1 Enhancement – Cross-Cutting Platform Capabilities: All 12 NDI domains now integrate with four platform-wide capabilities: Enhanced Search (dgop-search-service), Error Handling & Notifications (dgop-error-service + dgop-notification-service), Business Process & Workflow Management (dgop-bpm-service), and Cross-Domain Audit Trail & Monitoring. These capabilities are described in detail in Section 13.

1.3 Stewardship Role Taxonomy (v4.1)
1.3.1 Operational Data Steward (ODS)
The ODS is a front-line, point-of-entry data quality steward embedded in source systems and business processes. ODS represents "the most cost-effective form of DQ management" by preventing quality issues at the point of capture.
ODS Operational Lifecycle Integration:
DQ Phase 1 (Rule Definition): ODS validates rule feasibility at point of entry; flags impractical rules
DQ Phase 3 (Detection): ODS receives real-time alerts for entry-time violations; performs immediate correction
MCM Phase 6 (Matching): ODS reviews duplicate suspects at source before they enter matching engine
PDP Phase 3 (Consent): ODS ensures consent capture at data collection point
RMD Phase 3 (Value Governance): ODS uses authorized reference values only; reports unauthorized values
1.3.2 Project Data Steward (PDS)
The PDS is a temporary, project-specific steward assigned for the duration of data-intensive initiatives (migrations, system implementations, MDM launches).
PDS Operational Lifecycle Integration:
DG Phase 3 (Assign): PDS receives temporary ownership with project_scope flag; handover workflow triggered at project completion
DQ Phase 1 (Rule Definition): PDS defines project-specific DQ rules; rules reviewed for permanence at project closure
MCM Phase 3 (Golden Record Design): PDS designs match/merge rules for MDM launch projects
DAM Phase 4 (Physical Design): PDS ensures model lineage is maintained during migration
PDP Phase 5 (DPIA): PDS triggers DPIA for new systems processing personal data
1.3.3 Enterprise Data Steward (EDS)
The EDS is a senior community leadership role that coordinates across domains and drives program maturity.
1.4 Maturity Progression
All domains align with the unified NDI maturity scale:
Initial (1) – Ad-hoc, undocumented
Defined (2) – Documented, not consistently executed
Activated (3) – Operationalized with workflows and evidence
Enabled (4) – Automated, monitored, and continuously improved
Leading (5) – Predictive, AI-assisted, benchmarked
2. Domain 1: DG – Data Governance
2.1 Domain Overview
DG is the foundational NDI domain. It establishes ownership, stewardship, domain hierarchy, and accountability structures. Without DG operationalization, no other domain can achieve maturity.
v4.1 Enhancement: DG now includes Charter Management, Policy Lifecycle Management, Decision Timeframe governance, DGPO organizational sizing guidance, formalized Domain Council structures, 4-Tier Training curriculum, 3-Level Certification pathways (CDS/SDS/MDS), Community of Practice governance, and full integration with cross-cutting platform capabilities (Enhanced Search, Error Handling, Workflow Management, and Audit Trail).

3. Domain 2: MCM – Master Data Management
3.1 Domain Overview
MCM operationalizes the governance of master data entities, reference data sets, and golden records. It ensures that core enterprise data (customers, citizens, employees, suppliers, assets, locations) is consistent, authoritative, and governed across systems.
v4.1 Enhancement: MCM now includes detailed MDM matching algorithms (deterministic and probabilistic), survivorship rule frameworks, pre-configured templates for 7 master data domains, support for 4 MDM implementation styles (Registry, Consolidation, Coexistence, Centralized), and full integration with cross-cutting platform capabilities.

Trigger: SOR declared
Activity: Match rules (exact, fuzzy, probabilistic), merge rules (survivorship), and attribute lineage defined
Actor: Data Engineer + MCM Steward
v4.1 – Matching Algorithms:
Deterministic Matching:
Probabilistic Matching:
Match Resolution 5-Step Process: 1. Candidate Selection: Blocking/indexing to reduce comparison space (e.g., same country code, same first 3 characters of name) 2. Comparison: Apply match rules; generate similarity scores per field 3. Scoring: Aggregate field scores using configured weights; compute overall match probability 4. Classification: Auto-merge (above auto-threshold), manual review (between thresholds), no-match (below review threshold) 5. Resolution: Auto-merged records processed; review queue populated for manual steward review
v4.1 – Survivorship Rules:
Survivorship Rule Priority: Rules evaluated in priority order; first matching rule determines winning value; default to "most recent" if no rule matches.
Phase 4: Stewardship Assignment

Activity: MCM-specific roles assigned: MCM Owner, Entity Steward, Source System Steward
