Enhancement Summary (v4.1)
This enhanced version of Volume 4 addresses 15 critical capability gaps:
1. Solution Architecture Overview
1.1 Design Principles
Microservices Architecture: Domain-driven decomposition with independent deployability and scalability.
Cloud-Native with Sovereignty: Primary deployment on CITC-certified KSA cloud providers (STC Cloud, Mobily Nebula, Oracle Cloud Jeddah, Azure KSA) with guaranteed data residency.
API-First Design: All capabilities exposed via RESTful APIs with OpenAPI 3.1 contracts for national platform integration (GSB, Nafath, National Data Platform).
Event-Driven Integration: Apache Kafka for asynchronous, decoupled integration with external tools and internal services.
Zero Trust Security: NCA ECC-1:2024 aligned controls with identity-centric security, micro-segmentation, and continuous verification. ABAC-enhanced access control for Confidential data.
Bilingual by Design: Arabic (ar-SA) and English (en-US) first-class support in all interfaces, APIs, metadata, and documentation with RTL/LTR rendering.
Data Sovereignty: All production data and encryption keys remain within KSA; cross-border transfers require explicit DGC approval and PDPL safeguards.
Configuration over Code: Administrators configure domains, subjects, rules, workflows, KPIs, training tiers, certification paths, masking policies, and SDLC gates without developer intervention.
Privacy by Design: SDLC phase-gate integration ensures privacy requirements are embedded from Requirements through Deployment.
Master Data Governance: Native matching engine supports deterministic and probabilistic algorithms with configurable survivorship rules.
Unified Search Experience (NEW v4.1): Single search interface across all DGOP object types with faceting, auto-complete, and search analytics.
Intelligent Notification Delivery (NEW v4.1): Multi-channel notification system with template management, preference resolution, and delivery tracking.
Workflow Automation (NEW v4.1): Full BPMN 2.0 workflow engine with SLA monitoring, approval chains, RACI matrices, and tamper-evident audit trails.

2. Domain-Driven Design & Bounded Contexts
2.1 Domain Decomposition Strategy
DGOP is decomposed into bounded contexts aligned with NDI domains and functional modules. Each bounded context corresponds to a deployable microservice with its own domain model, database schema, and API surface. Cross-domain communication occurs exclusively through the integration layer (Kafka events) or the API gateway (REST calls).
The enhanced v4.1 architecture expands from 12 to 21 bounded contexts to address critical capability gaps in data profiling, master data matching, data masking, SDLC privacy integration, community of practice, certification management, advanced access control, unified search, multi-channel notifications, and business process management.
2.2 Bounded Contexts Mapping
2.3 Context Map & Integration Patterns
2.4 Anti-Corruption Layer (ACL)
The Integration Hub (dgop-integration-service) acts as the Anti-Corruption Layer for all external systems. It translates external domain models (Collibra, ServiceNow, Informatica, Nafath, Apache Griffin, Deequ) into the DGOP canonical model. No microservice communicates directly with an external system; all external integration is brokered through the Integration Hub.
The dgop-abac-service serves as a secondary ACL for authorization decisions, translating complex attribute-based policies into simple permit/deny responses for all upstream services.
The dgop-search-service (NEW v4.1) serves as a tertiary ACL for search queries, translating unified search requests into optimized Elasticsearch queries and enriching results from multiple domain contexts.
2.5 Shared Kernel
The following concepts are shared across bounded contexts via a shared library (dgop-common):
UUID v7 generation and validation
Bilingual text value object (ar-SA / en-US with RTL/LTR hints)
KSA Business Days calculator (Hijri + Gregorian holiday API)
Audit log entry schema (tamper-evident chain)
Event envelope schema (CloudEvents 1.0 compliant)
Security context (JWT claims, national_id, entity_code, roles, ABAC attributes)
Exception hierarchy (DGOP-001 through DGOP-028)
Masking utility functions (technique registry, format-preserving algorithms)
Profiling data types (statistical summary, histogram, pattern)
Matching data types (match score, cluster ID, survivorship rule)
Search query types (SearchRequest, SearchResponse, FacetResult) (NEW v4.1)
Notification types (NotificationChannel, DeliveryStatus, TemplateVariable) (NEW v4.1)
BPM data types (WorkflowStatus, TaskAction, ApprovalDecision, SLABreach) (NEW v4.1)

5. Integration Architecture
5.1 Integration Design Principles
Canonical Data Model: All external data is transformed into the DGOP canonical model before consumption by any microservice.
Event-Driven Decoupling: External system changes are published as Kafka events; microservices subscribe rather than poll.
API-First Contracts: Every integration has an OpenAPI 3.1 or AsyncAPI contract versioned and stored in Git.
Idempotency: All inbound operations are idempotent using deterministic keys (UUID v7 or composite natural keys).
Resilience: Circuit breakers, retries with exponential backoff, dead-letter queues, and fallback to cached data.
Security: All integrations use mTLS or OAuth2/OIDC; service accounts with scoped permissions; no hardcoded credentials.
Observability: Every integration call is traced (OpenTelemetry), logged (structured JSON), and metered (Prometheus metrics).

5.3 Connector Specifications
5.3.1 Data Catalog Connector
Inbound: Asset inventory, glossary terms, metadata completeness, lineage, classification, data elements. Outbound: Certified owner, certified steward, governance status, approval status, steward comments. Protocol: REST webhook + scheduled batch (CSV/Parquet); Kafka for real-time metadata changes. Authentication: OAuth2 client credentials (service account).
5.3.2 Data Quality (DQ) Tool Connector
Inbound: DQ rules, DQ scores, failed rules, DQ issues, trend reports. Outbound: DQ issue owner, remediation owner, approved exception, closure decision. Protocol: REST API polling (every 15 min) + Kafka for real-time alerts.
5.3.3 DLP Tool Connector
Inbound: DLP incidents, policy violations, severity, user/system involved, asset affected. Outbound: Assigned reviewer, investigation status, exception approval, remediation decision. Protocol: Webhook (real-time) + REST API for status updates.
5.3.4 ITSM Connector (ServiceNow / Jira)
Bidirectional sync: Incident and change management correlation with governance workflows. Protocol: REST API with OAuth2; scheduled sync every 5 minutes.
5.3.5 Open Data Portal Connector
Outbound: Dataset metadata, publication status, update frequency. Inbound: Downloads, usage statistics, publication confirmation. Protocol: REST API (Open Data Portal REST spec) + webhook callbacks.
5.3.6 FOI Channel Connector
Channels: Email (IMAP + NLP), Website forms (REST API), CRM webhook, Call Center (manual entry), Ticketing system integration. Auto-Classification: spaCy Arabic NLP extracts requester info, category, urgency, language. Protocol: IMAP polling (5 min) + REST webhook + CRM event bus.
5.3.7 IAM / SSO Connector
Nafath (National SSO): SAML2/OIDC federation via Keycloak custom plugin. Azure AD / Active Directory: LDAPS or OIDC for enterprise entities. ABAC Attribute Sync: SCIM 2.0 extensions for user attributes (department, clearance level, org unit) synchronized to ABAC attribute store. Protocol: SAML2/OIDC for browser flows; SCIM 2.0 for user provisioning; gRPC for attribute streaming to ABAC service.
5.3.8 SIEM Connector
Outbound: Structured audit events (CEF/LEEF format) to Splunk/QRadar. Includes ABAC decision audit trail, masking audit events, profiling anomaly alerts, match resolution audit, SDLC phase-gate decisions, BPM audit trail, search query audit, notification delivery audit. Protocol: Syslog over TLS (TCP 6514) or HTTPS REST API.
5.3.9 LMS Connector
Outbound: Training catalog sync, course enrollment, completion status, certification. Inbound: Training progress, assessment scores, completion certificates. Protocol: xAPI (Tin Can API) or SCORM 2004 via REST. 4-Tier Training Sync: Enhanced sync for tiered curriculum modules, prerequisite validation, blended delivery progress (instructor-led + e-learning + hands-on + peer mentoring).
5.3.10 Data Profiling Tool Connector (NEW)
Supported Tools: Apache Griffin 0.6+, Amazon Deequ 2.x, Great Expectations 0.18+, custom Spark profiling jobs.
Inbound: Profiling jobs (column statistics, histograms, pattern frequencies); Anomaly detection results (outlier records, distribution drift); DQ metrics (completeness, uniqueness, validity scores per dimension); Cross-column analysis (correlation matrices, functional dependencies); Schema drift alerts (column additions, type changes, constraint violations).
Outbound: Asset metadata for profiling scope definition; Classification flags for sensitivity-aware profiling; Masking configuration for pre-profiling data protection in non-production environments; DQ dimension thresholds for anomaly triggering.
Protocol: Apache Griffin: REST API polling (every 30 min) + Kafka for real-time alerts; Amazon Deequ: Spark job orchestration via REST + S3/MinIO for profile output files (Parquet); Great Expectations: Python SDK via REST gateway + Expectation Suite sync; Custom Spark: Livy/Spark Operator REST API for job submission; MinIO for result files.
Authentication: OAuth2 client credentials for REST APIs; Kerberos keytab for Spark cluster authentication; Service account with scoped permissions per profiling tool.
Canonical Data Model Mapping:
Resilience: Circuit breaker: 3 consecutive profiling job failures -> 5 min open; Fallback: Serve last-known profile from cache (Redis TTL: 24h); DLQ: Failed profiling events routed to dgop.error.dlq with tool-specific error context.
5.3.11 Search Integration Connector **(NEW v4.1)**
Purpose: Real-time and batch indexing of all DGOP objects into Elasticsearch/OpenSearch for unified search.
Inbound (Index Events): Asset created/updated/deleted events -> index update; Domain/Subject configuration changes -> index update; Workflow case/task transitions -> index update; NDI evidence submissions -> index update; Policy definition changes -> index update; Stewardship assignment changes -> index update; BDE/DQ rule changes -> index update; User/Org unit changes -> index update.
Outbound (Search Queries): Unified search results with faceting; Auto-complete suggestions; Advanced query with boolean operators; Search analytics data.
Protocol: Real-time indexing: Kafka consumer (dgop.search.index.events) -> Elasticsearch bulk indexing; Full reindex: Admin-triggered REST API -> PostgreSQL CDC (Debezium) -> Kafka -> Elasticsearch; Delta indexing: Change data capture -> incremental index update; Search query: REST API (/api/v1/search) -> Elasticsearch query -> result enrichment.
Authentication: Elasticsearch basic auth / API key; Kafka SASL_SSL.
Resilience: Circuit breaker: Elasticsearch unavailability -> degraded search (DB fallback); Index replication: 1 replica minimum, auto-expand to 2 during peak; Retry: 3 attempts with exponential backoff for failed index operations.
5.3.12 Notification Gateway Connector **(NEW v4.1)**
Purpose: Multi-channel notification delivery with template resolution and delivery tracking.
Channels: In-System: Direct database write to notification inbox; Email: SMTP relay (Postfix/Exchange 365) with TLS 1.3; SMS: SMPP gateway (Mobily/STC SMS APIs) with delivery receipts.
Inbound: Notification events from Kafka (dgop.notification.events); Direct API calls from services; Batch digest requests.
Outbound: Delivery status updates to Kafka (dgop.notification.delivery.events); Bounce/failure notifications; Delivery analytics.
Template Resolution Flow:
Event type -> template lookup (by template_code)
Variable substitution (Thymeleaf engine)
Bilingual content selection (ar-SA / en-US based on user preference)
Channel-specific formatting (HTML email, plain text SMS, JSON in-system)
Priority-based routing (LOW -> digest batch, NORMAL -> immediate, HIGH -> immediate + retry, CRITICAL -> all channels)
Resilience: Per-channel circuit breaker: 5 consecutive failures -> 2 min open; Retry: Email 3 attempts (5 min intervals), SMS 2 attempts (1 min intervals); DLQ: Undeliverable notifications -> dgop.notification.dlq with error context.
5.3.13 BPM Process Connector **(NEW v4.1)**
Purpose: Workflow template deployment, process instance execution, and cross-system task synchronization.
Inbound: Business events from Kafka (dgop.bpm.workflow.events); Workflow template deployments (BPMN XML upload); Task actions (assign, complete, delegate, approve, reject); SLA timer events from Redis.
Outbound: Workflow state changes to Kafka; Task assignment notifications to dgop-notification-service; SLA breach alerts to dgop-kpi-service; Audit events to SIEM and long-term storage.
Protocol: Workflow trigger: Business event -> Kafka -> dgop-bpm-service -> Camunda Zeebe -> process instance; Task assignment: Zeebe job worker -> task created -> notification sent; Approval chain: Sequential/parallel approval -> decision recorded -> next task activated; SLA monitoring: Timer events -> SLA check -> escalation if breached; Audit logging: Every workflow action -> audit event -> tamper-evident hash chain.
Resilience: Zeebe cluster: 3 brokers minimum, replication factor 3; Workflow state: Event-sourced, replayable from Kafka; Circuit breaker: Zeebe unavailability -> queue events, retry with backoff.

6. API Specifications
6.1 API Design Standards
Base URL: https://dgop.{entity}.gov.sa/api/v{version}/
Content-Type: application/json (UTF-8); Accept-Language: ar-SA or en-US; Content-Language in response
Authentication: Bearer {JWT} in Authorization header; JWT issued by Keycloak with Nafath claims; ABAC attributes embedded as JWT claims (department, clearance_level, org_unit)
Authorization: RBAC (role-based) + ABAC (attribute-based) for Confidential data access; RBAC-only for Public/Internal data; ABAC mandatory for Restricted/Confidential data operations
Idempotency: Idempotency-Key header (UUID) for POST/PUT/DELETE; 24-hour deduplication window in Redis
Pagination: Cursor-based (recommended) for large datasets; offset-based with max page_size=100 for simple lists
Filtering: RSQL/FIQL syntax (?filter=domain.code==EMPLOYEE;status==ACTIVE)
Sorting: ?sort=-created_at (descending), ?sort=+name (ascending)
Field Selection: ?fields=id,name,status (sparse fieldsets)
Error Format: RFC 7807 Problem Details { type, title, status, detail, instance, errors[] }
Rate Limiting: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset headers
Versioning: URL path versioning (/api/v1/, /api/v2/); backward compatibility maintained for 2 versions

7. Event-Driven Architecture
7.1 Event Bus Overview
DGOP v4.1 employs Apache Kafka 3.6 as the primary event backbone, implementing an event-driven architecture with the following characteristics:
Exactly-once semantics via idempotent producers and transactional message delivery
Schema governance via Confluent Schema Registry with Avro schemas and backward/forward compatibility policies
CloudEvents 1.0 specification compliance for all event envelopes
Multi-region replication for disaster recovery via MirrorMaker 2
Topic naming convention: {domain}.{service}.{event-type}.{version} (e.g., governance.asset.created.v1)
Consumer group strategy: One consumer group per service per topic to ensure ordered processing within partitions
Dead Letter Topic (DLT): {original-topic}.dlq for failed message handling with exponential backoff retry

8. Database Design
8.1 Design Principles
UUID v7 as primary key format for all tables (time-sortable, lexicographically ordered)
PostgreSQL 15+ as primary relational database with native partitioning support
Monthly RANGE partitioning for high-volume tables using pg_partman
Tenant isolation via row-level security (RLS) policies on tenant_id column
Temporal versioning via created_at / updated_at / version columns on all entities
Soft deletion via deleted_at timestamp (no physical deletion of governance records)
JSONB columns for flexible schema extensions and semi-structured data
Full-text search via GIN indexes on searchable text columns
Foreign key constraints for referential integrity with ON DELETE RESTRICT for governance entities
Check constraints for data validation at the database level

8.4.13 Audit Chain Verification Function
-- ============================================================ -- Stored procedure for hash chain verification -- Usage: SELECT verify_audit_chain('pi-018f5a3e-ab1d...'); -- ============================================================ CREATE OR REPLACE FUNCTION verify_audit_chain(p_process_instance_id VARCHAR(64)) RETURNS TABLE ( entry_id VARCHAR(64), sequence_number BIGINT, integrity_check VARCHAR(16), computed_hash VARCHAR(64), stored_hash VARCHAR(64), verification_message TEXT ) AS $$ DECLARE v_entry RECORD; v_previous_hash VARCHAR(64) := ''; v_computed_hash VARCHAR(64); v_data_to_hash TEXT; BEGIN FOR v_entry IN SELECT entry_id, sequence_number, action, actor_id, target_id, details, previous_hash, current_hash, timestamp FROM bpm_audit_trail WHERE process_instance_id = p_process_instance_id ORDER BY sequence_number ASC LOOP -- For genesis entry (sequence 1), previous_hash is the genesis hash IF v_entry.sequence_number = 1 THEN v_previous_hash := v_entry.previous_hash; END IF; -- Compute hash: SHA-256 of (action + actor_id + target_id + details::text + previous_hash + timestamp) v_data_to_hash := v_entry.action || '|' || v_entry.actor_id || '|' || COALESCE(v_entry.target_id, '') || '|' || v_entry.details::TEXT || '|' || v_previous_hash || '|' || v_entry.timestamp::TEXT; v_computed_hash := encode(digest(v_data_to_hash, 'sha256'), 'hex'); entry_id := v_entry.entry_id; sequence_number := v_entry.sequence_number; computed_hash := v_computed_hash; stored_hash := v_entry.current_hash; IF v_computed_hash = v_entry.current_hash THEN integrity_check := 'VALID'; verification_message := 'Hash chain intact'; -- Update previous_hash for next iteration v_previous_hash := v_entry.current_hash; ELSE integrity_check := 'BROKEN'; verification_message := 'Hash mismatch at sequence ' || v_entry.sequence_number || '. Expected: ' || v_computed_hash || ', Found: ' || v_entry.current_hash; -- Continue verification but mark chain as broken v_previous_hash := v_entry.current_hash; END IF; RETURN NEXT; END LOOP; RETURN; END; $$ LANGUAGE plpgsql; COMMENT ON FUNCTION verify_audit_chain IS 'Verifies the cryptographic hash chain integrity for a process instance audit trail';

All DGOP v4.1 microservices are deployed as containerized workloads on Kubernetes 1.28+ clusters across CITC-certified KSA cloud providers. The deployment architecture follows these principles:
Namespace isolation: Each service deployed in its own namespace with dedicated RBAC
Pod Security Standards: restricted profile enforced on all namespaces
Horizontal Pod Autoscaling (HPA): CPU and memory-based scaling with custom metrics
Pod Disruption Budgets (PDB): Minimum availability guarantees during cluster maintenance
Affinity rules: Anti-affinity for pod distribution, node affinity for workload placement
Resource quotas: CPU/memory limits per namespace with burst allowance
Health checks: Liveness (TCP), readiness (HTTP), and startup probes
Graceful shutdown: SIGTERM handling with configurable termination grace period

11. Disaster Recovery and Business Continuity
11.1 DR Architecture Overview
DGOP v4.1 implements a multi-region active-passive DR strategy across two KSA cloud regions (Riyadh primary, Jeddah DR), with the following characteristics:
RPO (Recovery Point Objective): 5 minutes for transactional data, 1 hour for audit logs
RTO (Recovery Time Objective): 15 minutes for critical services, 4 hours for non-critical
Data replication: Synchronous for PostgreSQL (within Riyadh AZs), asynchronous cross-region via pglogical
Kafka replication: MirrorMaker 2 with sync replication factor across regions
Elasticsearch: Cross-cluster replication (CCR) with auto-follow patterns
Zeebe: Exports to Elasticsearch for state recovery; process state rebuildable from event log

Notification Service DR Behavior:
During DR failover, undelivered notifications are preserved in Kafka (7-day retention)
Notification service in DR region consumes from replicated Kafka topics
In-system notifications may be briefly delayed during failover (< 2 min)
Email/SMS delivery continues via backup gateways with no interruption
11.3.3 dgop-bpm-service DR
Zeebe DR Recovery Procedure:
Zeebe state is event-sourced and recoverable from Elasticsearch export
On DR activation: Import ES indices into DR Zeebe cluster
Active process instances resume from last exported position
In-flight tasks (not yet exported) may need manual reconciliation (< 1%)
Audit trail hash chain integrity verified post-recovery

#!/bin/bash # DR recovery verification for BPM audit hash chain # Usage: ./verify-bpm-audit-recovery.sh <process-instance-id> PROCESS_INSTANCE_ID=$1 DR_DB_HOST="bpm-db-dr.jeddah.dgop.gov.sa" PRIMARY_HASH_FILE="/backups/last-primary-chain-hash.txt" echo "=== BPM Audit Chain Recovery Verification ===" echo "Process Instance: $PROCESS_INSTANCE_ID" echo "Timestamp: $(date -Iseconds)" # 1. Verify chain integrity in DR echo "[1/4] Verifying hash chain integrity in DR database..." psql -h $DR_DB_HOST -d dgop_bpm -t -c " SELECT COUNT(*) as total_entries, SUM(CASE WHEN integrity_verified = true THEN 1 ELSE 0 END) as verified, SUM(CASE WHEN integrity_verified = false THEN 1 ELSE 0 END) as broken FROM bpm_audit_trail WHERE process_instance_id = '$PROCESS_INSTANCE_ID'; " | while read total verified broken; do echo " Total entries: $total" echo " Verified: $verified" echo " Broken: $broken" if [ "$broken" -gt 0 ]; then echo " WARNING: Hash chain broken entries detected!" fi done # 2. Compare genesis hash with primary echo "[2/4] Comparing genesis hash with primary..." DR_GENESIS=$(psql -h $DR_DB_HOST -d dgop_bpm -t -c " SELECT chain_genesis_hash FROM bpm_process_instances WHERE process_instance_id = '$PROCESS_INSTANCE_ID'; " | xargs) if [ -f "$PRIMARY_HASH_FILE" ]; then PRIMARY_GENESIS=$(cat "$PRIMARY_HASH_FILE" | grep "$PROCESS_INSTANCE_ID" | cut -d' ' -f2) if [ "$DR_GENESIS" = "$PRIMARY_GENESIS" ]; then echo " Genesis hash MATCH: $DR_GENESIS" else echo " WARNING: Genesis hash MISMATCH!" echo " Primary: $PRIMARY_GENESIS" echo " DR: $DR_GENESIS" fi else echo " Primary hash file not available for comparison" fi # 3. Verify chain continuity echo "[3/4] Verifying chain continuity..." psql -h $DR_DB_HOST -d dgop_bpm -t -c " SELECT verify_audit_chain('$PROCESS_INSTANCE_ID'); " | grep -c "BROKEN" BROKEN_COUNT=$? if [ $BROKEN_COUNT -eq 0 ]; then echo " Chain continuity: VERIFIED (no breaks)" else echo " Chain continuity: BROKEN ENTRIES FOUND" fi # 4. Check signature validity echo "[4/4] Checking digital signatures..." psql -h $DR_DB_HOST -d dgop_bpm -t -c " SELECT COUNT(*) as total_signed, SUM(CASE WHEN digital_signature IS NOT NULL THEN 1 ELSE 0 END) as signed_entries FROM bpm_audit_trail WHERE process_instance_id = '$PROCESS_INSTANCE_ID'; " echo "=== Verification Complete ==="
