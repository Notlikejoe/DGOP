Content Summary
1. Introduction to Volume 5
Volume 5 of the DGOP v4.1 Master Specification defines the security architecture, regulatory compliance mappings, and audit framework for the Enterprise Data Governance Operations Platform. This volume translates the governance and operational requirements defined in Volumes 1-4 into concrete security controls, compliance evidence, and assurance mechanisms. It serves as the authoritative reference for security engineers, compliance officers, Data Protection Officers (DPO), and auditors responsible for certifying DGOP against KSA national standards.
1.1 Purpose
The purpose of this volume is to:
Map every NCA Essential Cybersecurity Control (ECC-1:2024) to specific DGOP architectural components, configurations, and operational procedures.
Map every PDPL 2023 requirement to DGOP functional modules, data flows, and consent management mechanisms.
Define the Zero Trust security model governing identity, policy enforcement, secret management, and service-to-service communication.
Establish an audit and logging architecture that provides tamper-evident evidence chains for NDI assessments, forensic investigations, and regulatory inquiries.
Specify four-level data classification, encryption, key management, ABAC policies, Privacy by Design gates, data masking services, and cross-border data transfer controls required for sovereign KSA government data.
Define enhanced search security controls (Section 5.6) ensuring that all search operations across DGOP data assets are encrypted, access-controlled, audited, and protected against sensitive data exposure.
Define secure error handling and notification architecture (Section 5.7) ensuring that error messages reveal no sensitive internal information, that all notifications are encrypted and audited, and that forensic tracing is possible without exposing internal identifiers.
Define workflow engine security controls (Section 8.5) ensuring that business process models, task assignments, case data, SLA enforcement, and sub-process execution meet the highest security standards.
1.2 Scope
This volume covers all security and compliance layers of DGOP, including:
Perimeter, network, application, data, identity, and monitoring security layers (Defense in Depth).
NCA ECC-1:2024 control families: Identity & Access Management (ECC-1), Network Security (ECC-2), Data Protection (ECC-3), Logging & Monitoring (ECC-4), Application Security (ECC-5), and Cloud Security (ECC-6).

Recurring Compliance Calendar: Automated scheduling and tracking of quarterly and annual stewardship compliance tasks.
Workflow Engine Security: BPMN model signing, task-level ABAC, case data encryption, tamper-proof SLA enforcement, and comprehensive workflow audit trails.

2. Regulatory & Compliance Framework Overview
Before detailing control mappings, this section establishes the regulatory context and the governance relationship between DGOP and KSA national standards. DGOP does not merely ‘support’ compliance; it operationalizes compliance by embedding controls into workflows, evidence management, and automated scoring.
2.1 SDAIA NDI Compliance Pillar
The NDI evaluates entities across three pillars: Data Maturity, Compliance, and Operational Excellence. Volume 5 directly addresses the Compliance pillar by providing:
Automated control verification: Security configurations are continuously checked against ECC baselines and PDPL requirements.
Evidence automation: Audit logs, access reviews, encryption certificates, and penetration test reports are collected as structured evidence linked to NDI specifications (e.g., DG.C.4.8, PDP.C.2.1).
Maturity scoring: Security and privacy controls are mapped to the 5-level NDI maturity model (Initial -> Defined -> Activated -> Enabled -> Leading).
2.2 DGA NORA Security Domain Alignment
Under DGA NORA, the Security Domain intersects with Business, Data, Application, Technology, and Beneficiary Experience domains. DGOP implements NORA security principles through:
Business Security: RACI-defined accountability for security decisions (e.g., Security Reviewer role in stewardship assignments).
Data Security: Classification-driven encryption, field-level PII protection, ABAC policies, data masking, Privacy by Design gates, and retention policies enforced via the Data Asset Governance Hub.
Application Security: OWASP ASVS Level 2 compliance, SAST/DAST in CI/CD, API Gateway security policies, and PbD SDLC gates.
Technology Security: Cloud-native security (CSPM), container security (Pod Security Standards), and infrastructure-as-code scanning.
Beneficiary Experience Security: Bilingual secure portals, accessibility (WCAG 2.1 AA), and privacy-by-design UX.
2.3 Risk-Based Compliance Approach
DGOP adopts a risk-based approach where control rigor scales with data sensitivity and criticality. The Asset Governance Hub (Volume 2) feeds classification, personal data flags, and criticality scores into the security engine, enabling dynamic control enforcement. For example:
A Public dataset with no personal data triggers basic access logging and standard TLS controls.
An Internal dataset triggers authentication (RBAC + Nafath SSO) and TLS 1.3 controls.
A Restricted dataset containing personal data triggers RBAC + encryption (field-level for PII), quarterly access review, and DLP monitoring.
A Confidential dataset containing sensitive personal data triggers ABAC + encryption + DLP (context-aware policies, HSM-backed keys, mandatory MFA, quarterly stewardship reviews, DPIA requirement, and data masking in non-production environments).
3. NCA ECC-1:2024 Control-by-Control Mapping
This section provides a comprehensive, control-by-control mapping between the National Cybersecurity Authority (NCA) Essential Cybersecurity Controls (ECC-1:2024) and DGOP architectural components, configurations, and operational procedures. Each control family is decomposed into specific DGOP services, policies, and evidence artifacts required for audit and NDI assessment.
3.1 Control Family: Identity & Access Management (ECC-1)
ECC-1 governs how users, services, and devices are identified, authenticated, authorized, and audited. DGOP implements ECC-1 through a federated identity architecture built on Keycloak, Nafath national SSO, RBAC/ABAC hybrid authorization, and fine-grained policy enforcement.
Control Mapping Detail – ECC-1-4 (Access Control): DGOP enforces ABAC policies through OPA sidecars deployed alongside each microservice. ABAC policies evaluate three dimensions of attributes:
Subject Attributes: user.roles, user.department, user.clearance, user.domain_id, user.location, user.device_trust_score
Resource Attributes: asset.classification, asset.domain_id, asset.owner_id, asset.contains_pii, asset.sensitivity_level
Environmental Conditions: request.time (business hours enforcement), request.source_ip (geo-location), request.device_type (managed vs unmanaged), request.encryption_level (TLS version)
Example Rego policy for ABAC asset access:
package dgop.asset.allow import future.keywords.if import future.keywords.in # Allow DATA_OWNER full access to their own assets regardless of classification allow if { input.user.roles[_] == "DATA_OWNER" input.asset.owner_id == input.user.person_id } # Allow DMO_ADMIN access up to Confidential classification during business hours allow if { input.user.roles[_] == "DMO_ADMIN" input.asset.classification in ["Public", "Internal", "Restricted", "Confidential"] tod := time.now_ns() [hour, _] := time.clock(tod) hour >= 7 hour <= 18 input.user.location in ["KSA", "APPROVED_REMOTE"] } # Confidential data requires ABAC evaluation with device trust allow if { input.asset.classification == "Confidential" input.user.device_trust_score >= 0.8 input.user.mfa_verified == true input.user.clearance == "CONFIDENTIAL" input.action in ["READ", "EXPORT_MASKED"] }
OPA evaluates these policies at the API Gateway (Kong) via the OPA plugin, ensuring centralized, version-controlled authorization with millisecond latency. Policy decisions are cached in Redis for 5 minutes to reduce redundant evaluation. ABAC decisions are logged with full attribute context for audit trail completeness.
3.2 Control Family: Network Security (ECC-2)
ECC-2 mandates network segmentation, traffic encryption, and denial of unauthorized communication. DGOP implements these controls through Istio service mesh, VPC isolation, and API Gateway policies.
3.3 Control Family: Data Protection (ECC-3)
ECC-3 addresses data confidentiality, integrity, and availability. DGOP implements a defense-in-depth data protection strategy aligned with the four-level business classification scheme, PDPL requirements, and sovereign data residency.
Data Residency & Sovereignty: All production data, backup data, and encryption keys remain within the Kingdom of Saudi Arabia. Primary database clusters are deployed in CITC-certified cloud regions (Riyadh or Jeddah). Cross-region replication is limited to KSA regions only. No cross-border data transfer occurs without explicit Data Governance Committee (DGC) approval, PDPL safeguards, and Standard Contractual Clauses (SCC) documented in DGOP workflow case FOI-TRANSFER-001.
3.4 Control Family: Logging & Monitoring (ECC-4)
ECC-4 requires comprehensive, tamper-evident logging and real-time security monitoring. DGOP implements a unified audit pipeline that captures application, database, network, and access events with cryptographic integrity verification.
3.5 Control Family: Application Security (ECC-5)
ECC-5 covers secure software development, vulnerability management, and runtime application protection. DGOP integrates security into the CI/CD pipeline and runtime environment, including Privacy by Design gates.
3.6 Control Family: Cloud Security (ECC-6)
ECC-6 addresses cloud-specific risks including container security, infrastructure drift, and multi-tenancy isolation. DGOP is deployed on CITC-certified KSA cloud providers (STC Cloud, Mobily Nebula, Azure KSA, Oracle Jeddah).
NDI Evidence Link: All ECC control implementations above generate evidence artifacts that are uploaded to the NDI Compliance Hub (Volume 2, Section 5.5) and linked to specifications such as DG.C.4.8 (Governance Structure), DG.C.5.2 (Security Controls), and PDP.C.3.1 (Data Protection). Maturity scoring automatically advances when evidence is approved and controls pass automated compliance checks.

4.2 PDPL Chapter 3: Data Subject Rights (Articles 10-18)
DGOP provides a self-service Data Subject Rights (DSR) portal and automated backend workflows to fulfill access, rectification, erasure, restriction, portability, and objection requests within statutory deadlines.
4.3 PDPL Chapter 4: Legal Basis & Consent (Articles 19-24)
DGOP implements granular consent management and legal basis tracking as core metadata dimensions attached to every data asset and processing activity.
Consent Management Architecture:
Consent Capture: Digital consent collected via web portal or API with timestamp, IP address, user-agent, geolocation (KSA-only), and purpose granularity. Consent versioned and stored in DGOP_CONSENT_REGISTRY table.
Consent Withdrawal: One-click withdrawal portal; automatic workflow triggers restriction/erasure evaluation; cascade to all dependent processing activities and third-party data sharing agreements.
Legal Basis Registry: Each DGOP_DATA_ASSET with personal_data_flag=true links to DGOP_LEGAL_BASIS table recording basis type (one of 6 bases), justification text, DPO approval, and expiry date.
Parental Consent: For minors (under 18), Nafath age verification triggers parental consent workflow; guardian identity verified via Absher; consent linked to minor’s national_id with guardian_id reference.
Consent Lifecycle State Machine: NOT_REQUESTED -> REQUESTED -> GRANTED -> ACTIVE -> WITHDRAWN -> EXPIRED. Each transition is logged with SHA-256 hash, previous hash, and chain ID in the audit trail, ensuring tamper-evident evidence for regulatory inspection. Consent refresh workflows trigger automatically 30 days before expiry.
4.4 PDPL Chapter 5: Data Protection Impact Assessment (DPIA) & RoPA
Articles 25-28 mandate DPIAs for high-risk processing and maintenance of Records of Processing Activities (RoPA). DGOP operationalizes these through dedicated workflow types and registry tables.
4.5 PDPL Chapter 6: Data Breach Notification (Articles 29-31)
DGOP implements an automated breach detection and notification workflow aligned with the 72-hour statutory notification requirement to SDAIA and, where high risk, direct communication to affected data subjects.
Breach Detection: Anomaly detection rules in SIEM (Splunk/QRadar) monitor for: unusual bulk access, off-hours privileged queries, DLP high-severity incidents, failed authentication spikes, and unauthorized cross-border transfer attempts.
Breach Triage: Automated workflow case created in dgop-workflow-service with case_type=BREACH; assigned to Privacy Steward, DPO, and Security Officer; severity calculated based on data volume, sensitivity, and affected subject count.
72-Hour Notification: If severity >= HIGH, automated notification package generated for SDAIA including: breach description, categories of data, approximate number of subjects, likely consequences, mitigation measures, DPO contact. Package submitted via SDAIA secure API with delivery confirmation.
Subject Notification: If high risk to rights and freedoms, personalized notifications (Arabic/English) sent via SMS, email, and portal alert within 72 hours; notification content reviewed by Legal Reviewer workflow task.
Breach Register: All breaches logged in DGOP_BREACH_REGISTER with: breach_id, detection_date, notification_date, severity, root_cause, remediation_status, affected_assets, affected_subjects_count, sdaia_reference_number. Register is immutable and auditable.
4.6 PDPL Chapter 7: Cross-Border Data Transfer (Articles 32-35)
DGOP enforces KSA data sovereignty and cross-border transfer restrictions through technical controls embedded in the infrastructure and data architecture layers.
Data Residency Verification: Quarterly automated scan verifies that all production PostgreSQL primary and standby nodes, Redis masters, Kafka brokers, MinIO buckets, and HSM instances are physically located within KSA CITC-certified data centers. Scan results are uploaded as NDI evidence under specification PDP.C.5.2 (Cross-Border Controls) with automatic maturity scoring.
4.7 PDPL Chapter 8: Controller & Processor Obligations (Articles 36-44)
DGOP supports both controller and processor governance models through multi-entity tenancy, processor agreement workflows, and joint controller registers.
Processor Registry: DGOP_PROCESSOR table records: processor_name, contract_reference, processing_purpose, data_categories, subprocessor_list, security_measures, DPO_contact, audit_date. Linked to DGOP_DATA_ASSET via processor_assignment.
Processor Agreement Workflow: Standard template with mandatory clauses (purpose limitation, subprocessor approval, audit rights, breach notification, return/deletion). Legal Reviewer and DPO approval required; annual review triggered by scheduler.
Joint Controller Arrangements: DGOP_JOINT_CONTROLLER table with responsibility allocation matrix; transparent notice to data subjects identifying each controller’s role; workflow ensures notice publication before processing begins.
DPO Appointment & Contact: DGOP_PERSON role_type=PRIVACY_STEWARD designated as DPO for entity; contact details published in privacy notice and RoPA; DPO performance tracked via Steward Score Formula (Volume 2, Section 7.12).
NDI Evidence Link: PDPL compliance evidence is mapped to NDI domain PDP (Privacy & Data Protection) specifications. Example mappings: PDP.C.1.1 (Legal Basis Registry) -> DGOP_LEGAL_BASIS table export; PDP.C.2.1 (Consent Management) -> Consent registry audit; PDP.C.3.1 (DPIA) -> DPIA workflow case closure; PDP.C.4.1 (Breach Notification) -> Breach register and SDAIA submission confirmation; PDP.C.5.1 (Cross-Border Transfer) -> Geo-fencing scan and SCC library.
5. Zero Trust Architecture
DGOP adopts a Zero Trust security model based on the principle: ‘Never trust, always verify.’ Every access request–whether from a user, service, device, or automated process–is authenticated, authorized, and encrypted before access is granted. Trust is continuously evaluated based on identity, context, behavior, and asset sensitivity. This section details the four pillars of DGOP Zero Trust: Identity & Access Management (IAM), Open Policy Agent (OPA) with full ABAC, HashiCorp Vault, and Mutual TLS (mTLS), plus the Privacy by Design SDLC integration module, enhanced search security, and secure error handling & notification architecture.
5.1 Identity & Access Management (IAM)
IAM is the foundation of DGOP Zero Trust. All identities–human and machine–are centrally managed, strongly authenticated, and granted least-privilege access based on dynamic context.
5.1.1 Identity Provider Architecture
Primary IdP: Keycloak 24 deployed in KSA cloud region with custom Nafath SAML2/OIDC federation plugin. Nafath is the mandatory authentication path for all government users, leveraging Absher and National ID verification.
Service Identity: All microservices authenticate to each other using SPIFFE/SPIRE or Kubernetes service accounts with Istio Citadel-issued X.509 certificates. Service identity is bound to the pod lifecycle and rotated every 24 hours.
External System Identity: Integration connectors (Catalog, DQ, DLP, ITSM) use OAuth2 client credentials with JWT assertions; client secrets stored in Vault and injected via External Secrets Operator.
5.1.2 Authentication Flow (Human User)
Step 1: User accesses DGOP portal -> Step 2: Redirect to Nafath (OIDC authorization code flow with PKCE) -> Step 3: Nafath authenticates via Absher/National ID + biometric/OTP -> Step 4: Nafath issues authorization code -> Step 5: DGOP backend exchanges code for ID token + access token (RS256) -> Step 6: Keycloak validates token signature and KSA claims (national_id, entity_code, verified_iqama) -> Step 7: DGOP creates session in Redis with 15-minute idle timeout, issues DGOP session cookie (HttpOnly, Secure, SameSite=Strict) -> Step 8: API Gateway enforces token validation on every request.
5.1.3 Token Structure & Claims
JWT Access Token (15-minute expiry, 7-day refresh):
{ "sub": "usr_<uuid>", "national_id": "10xxxxxxxx", "entity_code": "MOF", "roles": ["DATA_OWNER", "BUSINESS_STEWARD"], "departments": ["dom_xxx", "org_xxx"], "classification_clearance": "CONFIDENTIAL", "domain_id": "dom_finance", "location": "KSA", "device_trust_score": 0.92, "mfa_verified": true, "auth_time": 1718445600, "iat": 1718445600, "exp": 1718446500, "iss": "https://keycloak.entity.gov.sa/realms/dgop", "aud": "dgop-api" }
5.1.4 Role-Based Access Control (RBAC)
DGOP_ROLE_TYPE registry defines 12 core roles: SYSTEM_ADMIN, DMO_ADMIN, DATA_OWNER, BUSINESS_STEWARD, TECHNICAL_STEWARD, DQ_STEWARD, PRIVACY_STEWARD (DPO), SECURITY_REVIEWER, OD_OFFICER, FOI_OFFICER, AUDITOR, EXECUTIVE.
Role mapping table bridges Nafath/entity Active Directory groups to DGOP role_types. Supports custom claims for government entities with complex organizational hierarchies.
Role activation requires workflow approval for privileged roles (DMO_ADMIN, SYSTEM_ADMIN, SECURITY_REVIEWER): request -> manager approval -> DGC approval -> JIT activation with 4-hour session limit.
5.1.5 Multi-Factor Authentication (MFA)
Mandatory MFA for all users with access to Restricted or Confidential assets.
TOTP via Microsoft Authenticator or Google Authenticator (preferred); SMS OTP via CITC-approved providers (STC, Mobily, Zain) as fallback.
Adaptive MFA: Risk-based step-up triggered by anomalous location (non-KSA IP), new device fingerprint, off-hours access, or access to high-sensitivity assets. Risk score calculated by SIEM behavioral analytics.
MFA enrollment rate is a KPI tracked in dgop-kpi-service with target 100% for all non-Public asset stewards.
5.1.6 Role-Data Mapping Designer (NEW)
The Role-Data Mapping Designer enables Business Stewards to define and maintain role-to-data-domain relationships that feed into RBAC enforcement. This capability bridges the gap between business governance and technical access control, ensuring that access policies align with data stewardship assignments.
Purpose
In traditional RBAC systems, role-to-permission mappings are defined by IT administrators without direct business input. The Role-Data Mapping Designer empowers Business Stewards–who understand data domains, sensitivity, and business usage–to define which organizational roles should access which data domains, at what classification level, and with what operations.

5.2.4 OPA Deployment Architecture
OPA deployed as sidecar container in every application pod (Istio + OPA Envoy plugin) and as a standalone bundle server for centralized policy distribution.
Policy bundles stored in Git (GitOps); CI/CD pipeline (GitLab CI) validates Rego syntax, runs OPA unit tests, and pushes signed bundles to MinIO/S3. OPA sidecars pull bundles every 5 minutes with signature verification (ECDSA P-256).
OPA decisions logged to Kafka topic dgop.audit.opa with: decision_id, query, input (sanitized), result, latency_ms, policy_version, abac_attributes_evaluated.
5.2.5 Performance & Caching
OPA evaluation latency target: p99 < 5ms for authorization decisions.
Redis caches OPA decisions for 5 minutes (TTL) using cache key: opa:{policy_hash}:{user_id}:{resource_id}:{action}:{device_trust_score}. Cache invalidation triggered on role change, policy update, or device trust score change.
Partial evaluation pre-computes known static conditions (e.g., role-based allow rules) to reduce runtime query complexity.
For ABAC policies with environmental conditions (time, location), cache segments are time-bound and geo-bound to prevent stale decisions.
5.3 HashiCorp Vault
Vault is the central secrets management and cryptographic operations platform for DGOP. It provides dynamic secrets, encryption-as-a-service, PKI, and key management integrated with KSA Hardware Security Modules (HSM).
5.3.1 Vault Architecture
Deployment: Vault cluster deployed in auto-unseal mode with KSA HSM (Thales Luna 7 / Utimaco) via PKCS#11 integration. Seal shards split among 5 key custodians (CISO, DPO, DMO Lead, Security Engineer, Cloud Ops) with Shamir’s Secret Sharing (3-of-5 threshold).
High Availability: Active-passive cluster across Riyadh and Jeddah zones; Raft storage backend with synchronous replication; automatic failover via health checks.
Namespaces: Entity-isolated namespaces (vault/dgop/{entity_code}) ensure secret segregation between government entities sharing the platform.
5.3.2 Secret Types & Use Cases
5.3.3 Encryption-as-a-Service
Application-layer encryption: Microservices call Vault Transit API to encrypt/decrypt sensitive fields (national_id, mobile, email) before writing to PostgreSQL. This ensures database administrators and backup operators cannot read plaintext PII.
Key hierarchy: Data Encryption Keys (DEK) stored in Vault Transit -> Key Encryption Keys (KEK) stored in KSA HSM via PKCS#11. DEK rotation performed annually with re-encryption of historical data via background batch job.
Confidential data encryption: Separate DEK per data domain for Confidential assets; HSM-backed with PKCS#11; dual-control for key rotation (Security Engineer initiates, CISO approves).
Tokenization: Vault Transform engine replaces sensitive identifiers with format-preserving tokens (e.g., national_id 10xxxxxxxx -> 10****1234) for non-production environments and BI analytics.
5.3.4 Audit & Compliance
All Vault operations (authentication, secret access, policy change, key rotation) are streamed to Kafka topic dgop.audit.vault and forwarded to SIEM.
Vault policy changes require dual-control approval: Security Engineer submits pull request -> CISO approves -> automated deployment via GitLab CI -> OPA validates policy syntax before activation.
Quarterly HSM audit: HSM administrator generates audit log export; DGOP workflow case HSM-AUDIT-YYYY-QN routes log to Auditor role for review and NDI evidence upload.
5.4 Mutual TLS (mTLS)
mTLS ensures that every service-to-service communication within the DGOP Kubernetes cluster is cryptographically authenticated and encrypted. No service implicitly trusts another based on network location alone.
5.4.1 Istio Service Mesh mTLS
STRICT mode: All intra-cluster traffic uses mTLS with X.509 certificates issued by Istio Citadel (or cert-manager integration). PERMISSIVE mode is prohibited in production.
Certificate lifecycle: 24-hour validity; automatic rotation via Istio SDS (Secret Discovery Service); no manual certificate distribution. Revocation handled via Certificate Revocation List (CRL) distributed to Envoy proxies every 15 minutes.
Service identity: SPIFFE IDs encoded in certificates (spiffe://dgop.cluster.local/ns/dgop-{entity}/sa/{service-account}). OPA authorization policies reference SPIFFE IDs for service-to-service access control.
5.4.2 Traffic Policies
AuthorizationPolicy resources enforce allow-nothing default; explicit policies required for every service-to-service communication path. Example: dgop-asset-service can only accept traffic from dgop-workflow-service, dgop-ownership-service, and API Gateway.
Circuit breaker: Outlier detection (5 consecutive 5xx errors) ejects unhealthy pods for 30 seconds; retry policies limited to 3 attempts with exponential backoff to prevent amplification attacks.
Rate limiting: Local rate limit (50 req/s per pod) + global rate limit (1000 req/s per entity) enforced at Envoy sidecar; burst allowance for batch operations via API key tier.
5.4.3 East-West vs. North-South Traffic
North-South (external to cluster): TLS 1.3 termination at Istio Gateway (Kong -> Istio Gateway -> Service); WAF inspection at Kong; GeoIP filtering blocks non-KSA IPs for Confidential endpoints unless explicitly whitelisted via VPN bastion.
East-West (service to service): mTLS with SNI-based routing; no direct pod-to-pod communication bypassing sidecar; all traffic observable via Istio telemetry (Prometheus metrics, Jaeger traces, Envoy access logs).
Database traffic: Application pods connect to PostgreSQL via mTLS (client certificate authentication) with Vault-generated dynamic credentials; pgAudit captures all SQL statements with session correlation IDs.
5.4.4 mTLS Evidence for NDI
NDI specification ECC-2.2 (Traffic Encryption) and ECC-3.3 (Encryption in Transit) require evidence of mTLS enforcement. DGOP generates the following evidence artifacts automatically:
Weekly Istio configuration export (PeerAuthentication, DestinationRule, AuthorizationPolicy YAML) stored in evidence repository.
Monthly SSL Labs / testssl.sh scan report for external endpoints (TLS 1.3, certificate validity, cipher suite compliance).
Quarterly service mesh audit: manual review of mTLS policy coverage, certificate rotation logs, and revoked certificate list.
Real-time mTLS compliance KPI: percentage of services with STRICT mTLS (target 100%).

5.6 Enhanced Search Security (v4.1 NEW)
DGOP implements a comprehensive search security architecture that ensures all search operations–whether across data assets, business data elements, stewardship records, or governance documents–are cryptographically protected, access-controlled, audited, and safeguarded against sensitive data exposure. Search is a high-risk operation because it aggregates data across domains and classifications, potentially exposing information beyond a user’s authorized scope. This section defines the security controls governing all DGOP search capabilities, including the DGOP Navigator search interface, federated cross-entity search, and full-text indexing infrastructure.
5.6.1 Search Query Encryption
All search traffic is encrypted end-to-end using TLS 1.3, with additional protections at each layer of the search architecture.
Query Encryption for Sensitive Keywords: Search queries containing known sensitive keywords (national_id patterns, credit card sequences, health record identifiers) are flagged by the Query Sanitization Engine and encrypted with an additional layer of HSM-backed AES-256-GCM before being written to any log or index. The sanitized query is used for matching; the original query text is never persisted in plaintext.
5.6.2 Search Result Access Control (ABAC-Enforced)
DGOP enforces Attribute-Based Access Control on all search results, ensuring users only see documents and data assets for which they have explicit authorization. The search result filtering architecture operates as follows:
ABAC Search Policy Example:
package dgop.search.allow import future.keywords.if import future.keywords.in # DENY: Search across unauthorized domains deny if { input.search.scope.cross_domain == true not input.user.roles[_] == "DMO_ADMIN" not input.user.cross_domain_authorized == true } # DENY: Search for Confidential data without device trust deny if { input.search.target_classification == "Confidential" input.user.device_trust_score < 0.8 } # DENY: Search for Restricted data without MFA deny if { input.search.target_classification == "Restricted" input.user.mfa_verified != true } # ALLOW: Business Steward searches within their domain allow if { input.user.roles[_] == "BUSINESS_STEWARD" input.search.domain_id == input.user.domain_id input.search.target_classification in ["Public", "Internal", "Restricted"] input.user.mfa_verified == true } # ALLOW: Data Owner searches their own assets (all classifications) allow if { input.user.roles[_] == "DATA_OWNER" input.search.asset_owner_id == input.user.person_id input.user.mfa_verified == true input.user.device_trust_score >= 0.7 }

7. Audit & Logging Architecture
Audit and logging are the evidentiary backbone of DGOP compliance. Every governance action, security decision, data access, ABAC policy evaluation, masking operation, PbD gate transition, search query, error event, notification delivery, workflow decision, and authentication failure is recorded in a tamper-evident, immutable, and searchable log infrastructure. This architecture satisfies NCA ECC-4, PDPL accountability requirements, and SDAIA NDI evidence standards.
7.1 Audit Pipeline Architecture
The audit pipeline follows a multi-stage flow: Capture -> Enrich -> Route -> Store -> Analyze -> Retain. Each stage is designed for resilience, tenant isolation, and regulatory compliance.
Stage 1: Capture
Application Logs: Structured JSON logs from all microservices (Spring Boot Logback with ECS/NDJSON format) written to stdout; Fluent Bit DaemonSet collects from container filesystem with Kubernetes metadata enrichment.
Database Audit: PostgreSQL pgAudit extension captures all DDL, DML, and DCL statements with session-level and object-level granularity; audit logs streamed to dedicated Kafka topic dgop.audit.database via Debezium CDC.
Network Access: Istio Envoy access logs (HTTP/gRPC) captured via Istio telemetry API; includes source/destination SPIFFE IDs, mTLS status, response codes, latency, and request/response sizes (headers only, no body for PII protection).
Security Events: OPA decision logs (including ABAC attributes evaluated), Vault audit logs, Keycloak event logs, WAF alerts (ModSecurity), masking operation logs, PbD gate transitions, search query logs, error event logs, notification delivery logs, authentication failure logs, and SIEM correlation events aggregated via syslog-ng and Fluentd.
ABAC Audit: Every ABAC authorization decision logged with full attribute context (subject attributes, resource attributes, environmental conditions) for forensic reconstruction and compliance evidence.
Masking Audit: Every masking operation logged with technique used, field masked, classification, environment, requesting user/service, and data volume.
Search Audit: Every search query logged with actor, timestamp, query terms hash, filters applied, result counts, and ABAC scope decision.
Error Audit: Every error response logged with correlation ID, error type, sanitization status, and client exposure assessment.
Notification Audit: Every notification logged with recipient, channel, content hash, delivery confirmation, and encryption status.
Stage 2: Enrich
Fluentd/Fluent Bit parsers normalize heterogeneous log formats into canonical DGOP Audit Event Schema (Section 7.2).
Enrichment adds: Kubernetes pod labels, namespace (entity isolation), tenant_id (for multi-entity deployments), asset classification from DGOP_DATA_ASSET cache, geolocation (KSA-only validation), ABAC decision attributes, masking operation metadata, search query metadata, error classification, and notification channel metadata.
PII scrubbing: Sensitive fields in log messages (national_id, email, mobile) are tokenized using Vault Transform before routing to long-term storage. Raw values retained only in SIEM hot storage with 90-day TTL and access restricted to Auditor role.
Stage 3: Route
Kafka topic dgop.audit.events is partitioned by entity_id (tenant isolation) and retention policy (hot: 7 days, warm: 90 days, cold: 7 years).
Real-time consumers: SIEM (Splunk/QRadar) for alerting; Grafana Loki for operational dashboards; DGOP dgop-reporting-service for scheduled audit pack generation.
Batch consumers: Apache Spark job (every 15 minutes) aggregates audit events into TimescaleDB for KPI calculation, anomaly detection, and NDI evidence summary.
Stage 4: Store
Hot Storage (0-90 days): Elasticsearch 8.x cluster with ILM (Index Lifecycle Management); searchable by auditor, DPO, and security analyst; field-level RBAC prevents unauthorized access to sensitive log dimensions.
Warm Storage (90 days-1 year): MinIO/S3 object storage with versioning; objects transitioned to Infrequent Access tier after 90 days; checksum verification (SHA-256) on every upload.
Cold Storage (1-7 years): MinIO/S3 with Object Lock (WORM) in Compliance mode; retention period 7 years; no deletion allowed even by root/admin; legal hold capability for forensic investigations; stored in secondary KSA region (Jeddah) for disaster recovery.
Stage 5: Analyze
SIEM correlation rules detect: brute-force authentication, privilege escalation, off-hours sensitive data access, DLP policy violation, unauthorized cross-border transfer attempts, ABAC policy violations, anomalous masking operation volumes, anomalous API consumption patterns, unusual search query patterns, authentication failure spikes, and notification delivery anomalies.
ML-based anomaly detection (Phase 4 roadmap) identifies behavioral deviations: user accessing assets outside their domain, service account querying unusual data volumes, workflow task reassignment patterns indicating insider threat, unexpected masking/detokenization requests, search queries for data outside user’s scope, or repeated error responses suggesting reconnaissance.
Executive security dashboard (Grafana) displays: failed login attempts (last 24h), top 10 sensitive data access events, open DLP incidents, overdue security reviews, mTLS compliance percentage, ABAC decision latency, masking operation volume, search query volume and anomaly count, error rate by service, and notification delivery success rate.
Stage 6: Retain
NCA ECC-4 mandates 7-year retention for security audit logs. DGOP enforces this via S3 Object Lock Compliance mode with automated retention policy enforcement.
Quarterly retention audit: automated script verifies that no audit log object has retention period < required; results uploaded as NDI evidence under ECC-4.2 (Log Integrity).
Legal hold: DPO or CISO can place legal hold on specific audit events (e.g., breach investigation); hold overrides automatic expiration and requires manual release with dual approval.

8.5.8 Sub-Module Security Briefs
The DGOP workflow engine comprises 13 sub-modules, each with specific security controls.
1. Workflow Configuration Security
BPMN model repository encrypted at rest with AES-256-GCM
Model deployment requires dual-control approval (Security Reviewer + CISO for Confidential workflows)
Configuration changes audited with integrity hash
GitOps-based configuration synchronization with drift detection
2. Execution Security
Process engine runs in dedicated Kubernetes namespace with restricted Pod Security Standards
Workflow service account has least-privilege RBAC (no cluster-admin)
Process execution sandboxed: no file system access, restricted network egress via Istio AuthorizationPolicy
Groovy scripts (if any) executed in sandboxed environment with restricted classpath and no external library loading
3. Task Management Security
Task query API enforces ABAC: users only see tasks for their candidate groups and authorized domains
Task completion requires active JWT with valid MFA claim for Restricted/Confidential cases
Bulk task operations require DMO_ADMIN role with additional approval workflow
Task impersonation prohibited; administrative task reassignment logged with dual approval
4. Role & Responsibility Security
Workflow candidate groups mapped to DGOP_ROLE_TYPE registry; mapping changes trigger workflow re-deployment
Role-to-task bindings reviewed quarterly as part of access recertification
Emergency role assignment (e.g., temporary Security Reviewer coverage) requires JIT workflow with 24-hour auto-expiry
5. RACI Enforcement
BPMN models encode RACI accountability: Responsible (task assignee), Accountable (task approver), Consulted (commenter), Informed (notification recipient)
RACI violations detected by model scanner: tasks without accountable role, multiple accountable roles, circular consultation
RACI execution logged for compliance reporting
6. Approval Chain Security
Approval chains (sequential, parallel, conditional) defined in signed BPMN model
Approval chain modifications require model redeployment with dual approval
Override of approval chain requires: documented justification, Security Reviewer approval, CISO countersign, SIEM alert
Approval quorum enforced: minimum approvers required before case progression
7. Notification Security
Notification templates stored in encrypted repository; template changes require Security Reviewer approval
Notification rendering engine sanitizes all dynamic content (HTML escape, no script injection)
Notification classification verified before send: Confidential case notifications use encrypted channels only
Bilingual notification templates (Arabic/English) reviewed for security-appropriate messaging
8. SLA Enforcement Security
SLA definitions signed at model deployment; tamper detection via hash verification
SLA timers synchronized to KSA NTP; HSM-signed timestamps for deadline computation
SLA override workflow: justification -> Security Reviewer -> CISO -> SIEM alert -> signed override record
SLA compliance reports generated monthly for NDI evidence
9. Evidence Security
Workflow-generated evidence (approvals, decisions, certificates) signed with ECDSA P-256
Evidence storage: encrypted MinIO/S3 with WORM for 7 years
Evidence chain of custody: SHA-256 hash chain linking evidence to workflow instance
Evidence access: ABAC-enforced; Auditor role can view all evidence; Business Steward limited to their domain
10. Monitoring Security
Workflow engine metrics exposed via Prometheus with mTLS-only access
Workflow dashboards in Grafana require AUTHENTICATED role minimum
Security-critical workflow alerts (SLA breach, unauthorized deployment, task delegation anomaly) route to SIEM
Dashboard data filtered by entity and domain based on viewer’s authorization
11. Audit Trail Security
Complete BPMN execution trace logged to Kafka with integrity hash
Audit log schema includes: process_instance_id, activity_id, actor, timestamp, variable_hashes, decision_rationale
Audit logs immutable: WORM storage, 7-year retention, legal hold capability
Audit log access: AUDITOR role with ABAC filtering by classification and domain
12. Dashboard Security
Workflow dashboard data filtered by user’s domain_id and classification_clearance
Summary statistics aggregated to prevent individual case inference by unauthorized users
Dashboard export requires approval workflow for Restricted/Confidential content
Real-time dashboard updates via WebSocket Secure (WSS) with JWT authentication
13. Core Capability Security
Workflow engine API endpoints protected by Kong Gateway with OAuth2/OIDC validation
API rate limiting: 100 req/min per user for workflow operations; 10 req/min for workflow deployment
API input validation: strict schema validation for all request payloads; parameterized queries for database operations
Dependency security: Camunda engine dependencies scanned monthly (Trivy); CRITICAL vulnerabilities patched within 24 hours
NDI Evidence Link: Workflow engine security controls generate evidence artifacts uploaded to NDI Compliance Hub under specifications DG.C.4.8 (Governance Structure), DG.C.5.2 (Security Controls), ECC-1.5 (Privileged Access), and ECC-4.1 (Audit Logging). 9. Compliance Automation & Evidence Management
DGOP transforms compliance from a manual, periodic exercise into a continuous, automated, and evidence-driven operational process. This section describes how security and privacy controls are automatically verified, evidence is collected, NDI maturity is scored in real-time, and how recurring stewardship compliance tasks are scheduled, assigned, and tracked.
9.1 Automated Compliance Checking
Policy-as-Code: OPA policies (Rego) encode compliance rules (e.g., ‘all Restricted assets must have field-level encryption enabled’, ‘all Confidential assets must have active ABAC policies and HSM keys’, ‘all search queries must be logged with ABAC scope decision’, ‘all error responses must be sanitized before client exposure’, ‘all workflow models must be signed before deployment’). Policy violations are detected at deployment time (Terraform plan scanning, Kubernetes admission control) and runtime (API request evaluation).
Configuration Drift Detection: ArgoCD monitors Kubernetes manifests and Terraform state; drift from security baseline (e.g., mTLS disabled, privileged container deployed) triggers automatic remediation (GitOps self-healing) or alert for manual review if automated fix is unsafe.
Continuous Control Monitoring (CCM): Scheduled jobs (Kubernetes CronJobs) execute compliance checks every 6 hours: certificate validity, encryption key rotation status, access review overdue count, DLP incident open age, backup verification status, geo-fencing configuration, ABAC policy coverage, masking policy compliance, PbD gate completeness, search security policy compliance, error sanitization pass rate, notification encryption verification, workflow model signature coverage. Results written to DGOP_COMPLIANCE_CHECK table.
Vulnerability-Based Compliance: Trivy/Snyk scan results automatically update asset risk scores; assets with unresolved CRITICAL vulnerabilities are flagged non-compliant and excluded from Open Data candidate eligibility.
9.2 Evidence Collection & NDI Integration
Every security control, compliance check, and audit event generates structured evidence that is linked to NDI specifications, maturity levels, and assessment cycles. Evidence management is fully integrated with the NDI Compliance Hub (Volume 2, Section 5.5).
Evidence Types:
Configuration Evidence: YAML manifests, Terraform plans, policy bundles, certificate inventories, ABAC policy bundles, masking policy configurations, PbD gate certificates, search security configurations, error handling policy manifests, workflow model signatures (automated export).
Process Evidence: Workflow case closures, approval chains, access review completions, training certificates, role-data mapping recertifications, quarterly processing reviews, quarterly cross-border monitoring reports, search audit reviews, error sanitization tests, workflow security assessments (workflow-generated).
Technical Evidence: Penetration test reports, vulnerability scans, SIEM dashboards, DLP incident closures (integration-imported).
Audit Evidence: Audit log extracts, hash chain verification reports, HSM audit trails, ABAC decision audits, masking operation audits, search query audits, error event audits, notification delivery audits, workflow execution audits (pipeline-generated).
Evidence Lifecycle: DRAFT (auto-collected) -> SUBMITTED (steward review) -> UNDER_REVIEW (DMO/auditor validation) -> APPROVED (NDI scoring) -> EXPIRED (auto-detected by scheduler) -> REFRESH (re-collection triggered).
Maturity Scoring: Compliance check results feed the NDI readiness algorithm (Volume 4, Section 12.2). Example: If ECC-4.2 (Log Integrity) evidence is APPROVED and current, the PDP domain maturity score increases; if EXPIRED, score decreases and gap alert generated. New evidence types for ABAC (ECC-1.4), masking (ECC-3.4), PbD (PDPL Art. 25), search security (ECC-2.3), error handling (ECC-5.3), notification encryption (PDPL Art. 9), and workflow security (ECC-1.5) are linked to corresponding NDI specifications.
Audit Pack Generation: One-click export (PDF/A, XML, ZIP) for SDAIA assessors includes: all evidence files, workflow history, steward performance scorecards, compliance check summaries, recurring compliance calendar status, privacy steward responsibility completion report, and audit log extracts with chain-of-custody manifest.
9.3 Gap Analysis & Remediation Queue
The Compliance Automation engine performs automated gap analysis across NDI domains, PDPL requirements, and ECC controls:
Missing Evidence Detection: Compares NDI specification registry against approved evidence; generates prioritized gap list by domain, specification type (M > C1 > C2 > C3), and entity.
Expired Evidence Detection: Scans evidence expiry dates; alerts evidence owner 30 days before expiry; escalates to DMO Admin 7 days before expiry; auto-reduces maturity score on expiry.
Incomplete Workflow Detection: Identifies workflow cases stuck in approval for > SLA threshold; calculates bottleneck by role type (e.g., Security Reviewer backlog); feeds into Steward Performance Score penalty.
Unassigned Owner Detection: Scans DGOP_DATA_ASSET for missing owner/steward; creates ownership assignment workflow case with priority based on classification (Confidential > Restricted > Internal > Public).
ABAC Policy Gap Detection: Identifies Confidential assets without active ABAC policies; flags Restricted assets without MFA enforcement; generates prioritized remediation queue.
Masking Policy Gap Detection: Identifies non-production environments with unmasked Restricted/Confidential data; flags missing masking policies for newly classified assets.
PbD Gap Detection: Identifies active projects missing PbD review cases; flags production deployments without PbD certificates.
Search Security Gap Detection: Identifies search queries not being audited; flags Elasticsearch indices without field-level security; identifies missing rate limiting configuration.
Error Handling Gap Detection: Identifies services returning unsanitized error messages; flags missing RFC 7807 compliance; identifies error responses without correlation IDs.
Workflow Security Gap Detection: Identifies unsigned BPMN models in production; flags workflows without SLA tamper-proofing; identifies missing task-level ABAC policies.
Remediation Queue: All gaps are aggregated into a single prioritized queue visible in DMO Dashboard and ODIAO Cockpit; filtered by domain, severity, and due date; exportable to Excel/PDF for committee review.

Appendix F: ECC-to-DSP Activity Crosswalk (NEW)
This appendix maps each NCA ECC control to Data Stewardship Program (DSP) activities for unified compliance reporting. The crosswalk enables entities to report security control compliance and stewardship program compliance through a single unified framework.
Unified Compliance Score
The ECC-to-DSP crosswalk enables calculation of a Unified Compliance Score that represents both technical security control compliance and organizational stewardship compliance:
Unified Compliance Score = (ECC Technical Control Compliance * 50%) + (DSP Stewardship Activity Compliance * 50%)
Where: * ECC Technical Control Compliance = % of ECC controls with APPROVED evidence * DSP Stewardship Activity Compliance = % of DSP activities completed on time with evidence
This score is calculated per entity, per domain, and per quarter, and contributes to the NDI Compliance pillar maturity rating.

Term | Definition
ABAC | Attribute-Based Access Control – authorization based on user attributes, resource attributes, and environmental context (time, location, device).
BPMN | Business Process Model and Notation – standard graphical representation for specifying business processes in a workflow.
CCM | Continuous Control Monitoring – automated, real-time verification of security control effectiveness.
CSPM | Cloud Security Posture Management – continuous monitoring and remediation of cloud infrastructure misconfigurations.
DEK | Data Encryption Key – symmetric key used to encrypt data at rest, wrapped by KEK.
DGOP Navigator | The unified DGOP user interface framework that provides consistent navigation, search, discovery, and interaction patterns across all DGOP modules. Formerly referred to as “Design System,” DGOP Navigator encompasses the component library, search interfaces, notification center, and user experience patterns that enable users to securely navigate the platform.
DPIA | Data Protection Impact Assessment – systematic evaluation of privacy risks for high-risk processing.
DSR | Data Subject Rights – rights of individuals under PDPL (access, rectification, erasure, restriction, portability, objection).
ECC | Essential Cybersecurity Controls – NCA security control framework (ECC-1:2024).
HSM | Hardware Security Module – physical device for secure key generation, storage, and cryptographic operations.
JIT | Just-In-Time – temporary privilege elevation with time-bound scope and approval.
KEK | Key Encryption Key – key used to encrypt DEKs, stored in HSM.
LIA | Legitimate Interest Assessment – balancing test evaluating controller’s legitimate interest against data subject’s rights and freedoms.
mTLS | Mutual Transport Layer Security – bidirectional certificate-based authentication between services.
OPA | Open Policy Agent – policy-as-code engine for declarative authorization decisions.
